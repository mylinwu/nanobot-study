# 01｜项目全景与学习路线

## 1. 先用一句话理解 NanoBot

你可以把 NanoBot 理解为：

> 一个“会规划、会调用工具、会记忆、会自我修正”的智能体系统。

它不是只会聊天，而是会把目标拆解成步骤，并且把每一步交给最合适的能力模块完成。

---

## 2. 系统想解决什么问题？

### 典型需求

用户给一个复杂目标，例如：

- “帮我调研某技术方案，并输出可执行计划”
- “从一批文档里找证据，生成结构化结论”
- “自动执行多步骤任务：检索 → 分析 → 汇总 → 复核”

### 最朴素实现（为什么不够）

最简单做法：把全部信息丢给一个大模型，让它一次性回答。

问题：

1. 不可控：步骤不透明，不知道它怎么得出结果。
2. 不稳定：同样问题多次结果波动较大。
3. 不可扩展：难接入检索、数据库、外部 API、代码执行等工具。
4. 不可审计：出现错误时难定位是哪一步错了。

---

## 3. NanoBot 这类架构的核心思想

1. **任务分层**：目标层（Goal）→ 计划层（Plan）→ 执行层（Act）。
2. **能力解耦**：规划、执行、记忆、检索、反思、输出分别负责。
3. **闭环迭代**：执行后评估，失败则修正策略继续执行。
4. **可观测性**：全流程日志、步骤状态、调用痕迹可追踪。

---

## 4. 建议学习路径（渐进式）

- 第一步：先看“一个任务怎么跑完”。
- 第二步：再看“每个模块为什么存在”。
- 第三步：最后看“如何自己实现并工程化”。

你在这套文档中的学习顺序，就是按这个路径设计的。
