# 08｜部署、监控、评测与成本优化

> 本章不是“运维名词罗列”，而是告诉你：为什么同一个 Agent 在 Demo 里很好、上线后却经常翻车。

## 1. 为什么工程化阶段会出问题？

常见现象：

- 本地测试通过，上线后超时暴增。
- 任务能跑完，但成本失控。
- 同一请求结果波动太大，业务不敢用。

根因通常不是“模型不够强”，而是：

1. 调度策略不匹配流量。
2. 没有建立可观测链路。
3. 没有做质量与成本的双指标治理。

---

## 2. 渐进式推演：部署架构（V1→V4）

### V1：单体服务

- API、调度、执行都在一个进程。
- 优点：快，适合 PoC。
- 问题：高峰时互相拖慢，故障隔离差。

### V2：控制面/执行面拆分

- 控制面负责计划与状态。
- 执行面负责工具调用。

收益：职责清晰，扩容更灵活。

### V3：任务队列化 + Worker 池

- 请求入队，worker 异步处理。
- 可按任务类型分队列（检索型、计算型、长任务）。

收益：削峰填谷，系统稳定性提升。

### V4：多租户与优先级调度

- 关键租户高优先级。
- 支持配额、隔离、限流。

收益：能支撑真实业务场景。

---

## 3. 监控指标（必须有）

### 3.1 结果质量指标

- 任务成功率
- 引用命中率（答案引用是否真的支持结论）
- 人工复核通过率

### 3.2 性能指标

- 平均时延、P95 时延
- 队列等待时长
- 工具调用超时率

### 3.3 成本指标

- 单任务 token 成本
- 单任务工具成本
- 失败重试附加成本

> 教学建议：先把“成功率、P95、单任务成本”这 3 个指标跑通，再扩展其他指标。

---

## 4. 渐进式推演：评测体系（V1→V4）

### V1：人工主观评估

问题：慢、主观、不可复现。

### V2：固定评测集（离线）

- 准确性
- 完整性
- 格式合规

### V3：线上抽样评测

- 真实流量中抽样复核。
- 关注“离线好、线上差”的偏移。

### V4：自动回归评测 + 发布门禁

- 每次改 Prompt/模型/策略都跑回归。
- 不达标不发布。

---

## 5. 成本优化（从最容易见效开始）

1. **模型分级路由**：简单任务走低成本模型。
2. **检索前置**：用证据缩小上下文，减少无效 token。
3. **结果缓存**：高频问题命中缓存。
4. **重试治理**：避免无限重试导致成本雪崩。

---

## 6. 章节练习（建议实际做）

请你为自己的 Agent 写一个最小“上线看板”，至少包含：

- 成功率（按日）
- P95 时延
- 单任务平均成本
- Top3 失败原因

如果你能画出这个看板，说明你已经具备基本工程化思维。
