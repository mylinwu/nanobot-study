# 07｜渐进式实现：从最小可用到工程化

> 这一章给你一条“可直接照着做”的落地路径。每一阶段都包含：目标、最小代码骨架、上线前检查点。

## 阶段 0：单轮问答（1~2 天）

**目标**：跑通基本请求响应。

**骨架**：

```python
@app.post('/chat')
def chat(q):
    return llm.generate(q)
```

**检查点**：

- 能稳定返回
- 有基本日志

---

## 阶段 1：加入 Planner + 线性执行（3~5 天）

**目标**：复杂任务可拆解。

**骨架**：

```python
plan = planner.make_plan(goal)
for step in plan:
    out = executor.run(step)
    store(step, out)
```

**检查点**：

- 每步可追踪（step_id）
- 失败可定位到具体步骤

---

## 阶段 2：引入工具注册中心（5~7 天）

**目标**：从“会说”升级为“能做”。

**新增**：

- search/sql/python 工具
- 统一输入输出 schema
- timeout/retry

**检查点**：

- 工具失败不会拖垮全链路
- 有审计日志（工具、参数、耗时）

---

## 阶段 3：加入记忆 + RAG（1~2 周）

**目标**：答案更可靠，减少幻觉。

**新增**：

- session summary memory
- vector index + rerank
- 引用证据 ID

**检查点**：

- 关键结论可回溯证据
- 检索失败有明确降级输出

---

## 阶段 4：加入反思与重规划（1 周）

**目标**：提高复杂任务成功率。

**新增**：

- Reflector 质量评估
- replan trigger（证据不足/工具持续失败）

**检查点**：

- 同任务多次运行结果波动收敛
- 失败重跑成本可控

---

## 阶段 5：生产化（持续迭代）

**目标**：可上线、可运维、可审计。

**必须建设**：

- tracing + metrics + alert
- 成本面板（token/tool）
- 权限隔离与安全护栏

**上线门槛建议**：

- 任务成功率 >= 90%
- P95 时延达到业务目标
- 单任务成本有可解释上限

---

## 附：四个模块的“至少三版”演进总表

| 模块 | V1 | V2 | V3 | V4 |
|---|---|---|---|---|
| Planner | 单步 | 线性多步 | DAG 依赖 | 自适应重规划 |
| Orchestrator | 串行 | 状态机+重试 | 并行+幂等 | 断点恢复 |
| Executor | 纯 LLM | 单工具 | 工具注册中心 | 容错/降级/审计 |
| Memory/RAG | 对话拼接 | 摘要+混合检索 | 重排+证据评分 | 无证据不下结论 |
