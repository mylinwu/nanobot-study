# 05｜模块拆解：记忆、知识与检索

## 1. 先从最小需求开始

需求：“让 Agent 不要每一步都失忆，并且能引用外部证据。”

---

## 2. 渐进式推演：记忆系统

### V1：会话内拼接历史

把最近几轮对话直接拼入 prompt。

- 优点：实现最简单。
- 问题：上下文很快爆 token；噪音高。

### V2：短期记忆摘要

每执行若干步，做一次摘要写入 `session_memory`。

```json
{"session_id":"x","summary":"用户关注成本和稳定性，偏好表格输出"}
```

- 收益：减少 token。
- 问题：摘要失真风险。

### V3：长期记忆与偏好画像

新增 `user_profile_memory`：

- 输出偏好（表格/分点）
- 风格偏好（简洁/详细）
- 业务上下文（行业/角色）

读取策略：只在相关任务注入，避免污染。

### V4：可控记忆（TTL + 优先级 + 可解释）

- TTL：过期自动失效。
- 优先级：任务内事实 > 长期偏好。
- 可解释：记录“本次注入了哪些记忆”。

---

## 3. 渐进式推演：RAG 检索

### V1：朴素向量检索

`query -> embedding -> topK`

问题：召回相关但不一定可回答。

### V2：查询改写 + 混合检索

- query rewrite（补关键词）
- dense + sparse 混合召回

问题：召回量上来后噪音也上来。

### V3：重排 + 证据打分

- Cross-encoder 重排
- 证据质量分（新鲜度/权威性/相关性）

### V4：答案约束（无证据不下结论）

生成时强制：

- 每条关键结论绑定证据 ID。
- 无足够证据时输出“不确定 + 缺失信息”。

---

## 4. 检索层最关键的工程指标

- Recall@K
- MRR / nDCG
- 引用命中率（答案中的引用是否真实支持结论）
- 每次检索 token 成本
