# 09｜源码对照阅读清单（实战）

> 目标：把“看文档懂了”变成“看源码也能定位到实现”。

## 1. 先建立阅读策略（很重要）

不要上来就从头到尾读代码，建议按这条路径：

1. 先找入口（请求从哪里进来）。
2. 再找主链路（任务如何流转）。
3. 再看模块细节（计划、执行、记忆、评估）。
4. 最后看异常路径（重试、降级、重规划）。

这样做的好处：你不会在细节里迷路。

---

## 2. A. 入口层（你必须先搞清楚）

先找：

- Web/API 入口
- CLI 入口
- 主流程启动点

你要回答的问题：

1. 用户请求在哪被接收？
2. 请求如何进入 Orchestrator？
3. 上下文对象如何初始化？

**验收标准**：你能画出“入口到调度器”的 10 行以内时序。

---

## 3. B. 计划层（Planner）

重点看：

- Planner 的输入输出结构
- 步骤依赖关系表达
- 失败后是否支持重规划

**建议你记录的字段**：

- `goal`
- `steps[].id`
- `steps[].depends_on`
- `steps[].done_when`

**验收标准**：你能给出一个真实 plan JSON 样例并解释每个字段用途。

---

## 4. C. 执行层（Executor/Tool）

重点看：

- Executor 调度策略（串行/并行）
- 工具注册与发现机制
- 错误分类（可重试/不可重试）

**重点排查**：

- 工具超时如何处理？
- 是否有 fallback 工具？
- 工具输出是否 schema 校验？

**验收标准**：你能追踪一次工具失败后的完整回路。

---

## 5. D. 记忆与知识层（Memory/RAG）

重点看：

- 会话记忆存储位置
- 检索服务接口
- 引用证据如何注入最终答案

**建议追踪一次检索链路**：

`query -> recall -> rerank -> evidence -> answer`

**验收标准**：你能拿一个结论反查到它对应证据。

---

## 6. E. 观测与运维层

重点看：

- 日志字段是否足够追踪一次任务
- 是否有 tracing / metrics
- 是否可按 task_id 回放执行过程

**验收标准**：你能用 task_id 回答“这个任务慢在哪、贵在哪、失败在哪”。

## 8. 学习提醒

你读源码时，最有价值的不是“看懂每一行”，而是：

- 看懂主链路；
- 看懂关键输入输出；
- 看懂失败如何被系统消化。

做到这三点，基本就能独立维护一个 Agent 系统。
