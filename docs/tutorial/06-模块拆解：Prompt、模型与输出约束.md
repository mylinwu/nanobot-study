# 06｜模块拆解：Prompt、模型与输出约束

## 1. 渐进式推演：Prompt 工程

### V1：单段 Prompt

把任务描述直接给模型。

问题：规则易丢失，复用性差。

### V2：三层 Prompt（System/Task/Step）

- System：全局规则（安全、风格、工具边界）
- Task：本次目标
- Step：当前步骤动作

收益：可维护性大幅提升。

### V3：模板化 Prompt + 变量注入

```text
[ROLE]{role}
[GOAL]{goal}
[CONTEXT]{context}
[TOOLS]{tool_specs}
[FORMAT]{output_schema}
```

收益：版本化管理，AB 测试更方便。

### V4：自适应 Prompt（基于错误反馈）

- 若格式错误频发：强化 schema 提示。
- 若幻觉频发：强化“先证据后结论”。

---

## 2. 渐进式推演：模型路由

### V1：单模型全流程

问题：成本高，且不一定适合每种子任务。

### V2：双模型路由

- 规划/反思：高推理模型
- 抽取/改写：轻量模型

### V3：按步骤类型路由

根据 `step.type` 路由：`planning / coding / extraction / writing`。

### V4：动态路由（质量-成本闭环）

- 先用低成本模型。
- 质量评估不过再升级模型重跑。

---

## 3. 渐进式推演：输出约束

### V1：自然语言输出

问题：上游难解析。

### V2：JSON 输出

问题：仍会出现字段缺失/类型错。

### V3：JSON Schema 校验 + 自动修复

- 校验失败时触发 `repair_prompt`。
- 限次修复，避免死循环。

### V4：结构化协议 + 强类型消费

- 函数调用/tool calling。
- 下游直接按强类型对象消费，减少脆弱解析。
